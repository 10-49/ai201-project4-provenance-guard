# Provenance Guard

Provenance Guard is a backend API designed to classify text as either human-written or AI generated by using a multi signal detection pipeline to evaluate
submitted content, calculate a confidence score, and assign to it a transparency label. It additionally includes an appeal system for submittors/creators to contest classifications.


## Architecture Overview

When a piece of text enters the system via the `POST/submit` endpoint, it's passed to the multi signal detection to be handled. Signal one is a groq llm which evaluates semantic meaning and tone. Signal two is a script that calculates structural heuristics like word length variance and vocab diversity.

Raw outputs from both of the signals are combined into a final AI score using a weighted average. This score is then mapped to a calibrated confidence percentage and a specific transparency label. The entire process is recorded into a local JSON audit log. When a user contests a transparency label they can hit the `POST/appeal` endpoint, which searches for their submission in the logs and updates the status of it to "under review"

## Detection Signals

The system uses two signals to analyze text:
1. **Semantic Analysis (Groq Llama-3.3-70b):** 
   * **What it measures:** Semantic patterns, predictability, and uniform tone. 
   * **Why I chose it:** AI models write in a predictable, safe, and specific way, showcasing patterns in writing not found in natural human writings.. 
   * **Blind spot:** Struggles with highly formal human writing, like academic research, which shares the rigid structure of AI text.

2. **Stylometric Heuristics (Python):**
   * **What it measures:** Sentence length variance, word length variance, and vocabulary diversity using Herdan's C.
   * **Why I chose it:** human writing is typically more erratic than AI writing - AI is uniform. 
   * **Blind spot:** Shorter (and especially very short) text submissions lack enough data points to create a meaningful standard deviation, can skew the math.

## Confidence Scoring

To protect human writers from false positives, the scoring does not treat both signals exactly equally. The formula weights the LLM heavier than the heuristics: 
`Final AI Score = (LLM Score * 0.65) + (Heuristic Score * 0.35)`
Another reason for the weighting is because of the heuristics dominating impact on scores during early on testing, though later on we will see how it was addressed. 

The final score scores the text into tiers: Likely Human (0.0-0.35), Uncertain (0.35-0.65), and Likely AI (0.65-1.0). The confidence percentage scales based on how deep the score is within its tier.

**Example 1: High-Confidence AI**
*   **Input text:** "Artificial intelligence represents a transformative paradigm shift in modern society..."
*   **LLM Score:** 0.95
*   **Heuristic Score:** 0.5
*   **Final AI Score:** 0.7925
*   **Label Result:** Generated by an AI (Confidence: 40.7%)

**Example 2: Lower-Confidence (Borderline Human)**
*   **Input text:** "I've been thinking a lot about remote work lately. There are genuine tradeoffs..."
*   **LLM Score:** 0.2
*   **Heuristic Score:** 0.257
*   **Final AI Score:** 0.220
*   **Label Result:** Written by a Human (Confidence: 37.1%)

## Transparency Labels

The system returns one of the strings to the client:

*   **High-Confidence Human:** "Written by a Human (Confidence: [X]%) — This content matches normal human writing patterns, showing natural stylistic and structural variety."
*   **Uncertain:** "Uncertain — The system detected mixed signals. This text contains a blend of highly uniform structures and personal stylistic variations. Reader discretion advised."
*   **High-Confidence AI:** "Generated by an AI (Confidence: [X]%) — This text exhibits heavy statistical markers, structural uniformity, and phrasing patterns typical of automated language models."

## Rate Limiting

The `/submit` endpoint is setup to be monitored and limited by flask-limiter. 
*   **Limits:** 10 requests per minute; 100 requests per day.
*   **Reasoning:** it is strict enough to stop a malicious script from flooding the server with fake submissions or draining Groq API credits while acting within the room for normal human writer uploads.

## Known Limitations 

The system will likely misclassify short poetry. A poem heavily relying on repetition, short stanzas, and a limited vocabulary will cause the signals to register zero sentence variance and low vocabulary diversity. This spikes the heuristic score, dragging human work into  "Uncertain" or "AI" tier.
Alongside this, the system definetly also struggles the shorter a text is for the same reason outlined for short poetry difficulties. 

## Spec Reflection
How the spec helped: having the exact JSON and label strings written out before I started my coding made building the Flask routes faster. I didn't have to think about what the API would return since the outline allowed me to look back and implement what I wrote

Where I diverged: In the spec, I was planning to use a standard type token ratio for the heuristic signal. When testing, I spent a significant amount of time in milestone 4 to realize the ttr broke on short paragraphs because many words were unique, dragging scores into the mud close to zero for some text while others were seemingly unaffected. I diverged from the spec by changing signal weighting, and especially in the heuristic department, upgrading the math to Herdan's C (logarithmic ttr) and adding word-length variance to handle smaller sample sizes. It somewhat resolved the issue but isn't a full solution. 

## AI Usage
Drafting the API Skeleton: I gave an AI my planning document and asked it to generate the basic Flask boilerplate and the Groq LLM integration function as intended. It produced a solid baseline, but terminal testing commands were non-functioning because unix commands werent working in my powershell terminal. I also asked for help regarding translating syntax from unix to powershell. 

Fixing the Statistical Math: When my heuristic signal kept failing on short text inputs and weight changes weren't enough, I prompted the AI to help me debug and fix the math. It provided solutions such as further weight changes and a hardcoded word count bypass (if a text was too small, heuristics would be ignored and set to a default .5) but I didn't like these solutions and revised my prompts. It then suggested the Herdan's C implementation to normalize the vocab diversity curve and minor updates to the other math. The implementations fixed the scaling issue.
